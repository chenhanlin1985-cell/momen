# 事件重新归类方案

本文档按新的游戏化分类方式整理当前事件，不再沿用旧的 `fixed_story / conditional_story / random_filler` 或上一版“职责型分类”。

新的一级分类统一采用 5 类：

1. 对话战斗事件
2. 随机事件
3. 奖励事件
4. 回顾事件
5. 商店事件

其中：

- 对话战斗事件再细分为：普通战斗 / 精英战斗 / BOSS 战斗
- 随机事件包含正向收益和负面收益，并可带检定
- 奖励事件是纯发放收益，不承担复杂风险
- 回顾事件负责恢复、整理、升级牌组
- 商店事件负责卡组调整与购买

这份文档只做两件事：

1. 把当前已经存在的事件归类
2. 标出当前还缺哪些类型，后续应该怎么补

---

## 1. 现有事件总览

当前 Act 1 里的事件，如果按你现在要的分类方式看，本质上分布是：

- 对话战斗事件：已经成型，是当前最清晰的一类
- 随机事件：已经有一批，但“检定后正负收益分流”还不够统一
- 奖励事件：有雏形，但很多还混在对话事件里
- 回顾事件：几乎还没正式做出来
- 商店事件：当前完全没有

也就是说：

当前项目最成熟的是“对话 -> 入侵 -> 心战 -> 战后收束”。
最缺的是“回顾事件”和“商店事件”。

---

## 2. 对话战斗事件

定义：

- 先有人物对话
- 玩家可选择 `观察 / 入侵`
- `入侵` 进入心战
- 战后进入胜败收束对话

这是当前最符合现有项目核心体验的一类。

### 2.1 BOSS 战斗

定义：

- 对应关键人物
- 直接决定路线是否成立
- 有明确战前铺垫、战中弱点、战后分支

当前归类如下：

- `2001 -> 9101 -> 9102 / 9103`
  - 柳飞霞首战
  - 当前应归为 `BOSS 战斗`

理由：

- 是第一场真正的核心心战
- 胜负影响后续人物线成立
- 还承担“心战系统教学”职责

### 2.2 精英战斗

定义：

- 重要人物线上的关键战
- 不一定直接终局，但会显著改变路线

当前归类如下：

- `2004 -> 9201 -> 9202 / 9203`
  - 王麻子线首战
  - 归为 `精英战斗`

- `2005 -> 9301 -> 9302 / 9303`
  - 夜巡弟子线首战
  - 归为 `精英战斗`

- `2003 -> 9401 -> 9402 / 9403`
  - 疯长老线首战
  - 归为 `精英战斗`

理由：

- 都是关键人物战
- 胜负会影响路线深度
- 但目前还没有做到柳飞霞那种“首个核心样板 BOSS”的戏份强度

### 2.3 普通战斗

定义：

- 刷经验
- 练牌组
- 练熟规则
- 不承担重要叙事转折

当前归类如下：

- `9501 -> 9502 / 9503`
  - 外门低级弟子杂兵战
  - 归为 `普通战斗`

说明：

- 这类是当前唯一真正成型的普通战斗
- 后续应继续扩充成一个稳定池

### 2.4 当前缺口

当前对话战斗事件还缺：

- 第二个 BOSS 战
- 更明确的精英战梯度
- 普通战斗的稳定刷新池

建议：

- 柳飞霞第二次深层交锋，做成第二个 `BOSS 战斗`
- 王麻子、疯长老保留为 `精英战斗`
- 低级弟子、血役、替班巡夜补成 `普通战斗池`

## 3. 随机事件

定义：

- 纯随机出现
- 结果可能正向也可能负向
- 理想状态下应尽量带检定或条件分流

当前项目里，这一类主要对应 `random_filler`。

### 3.1 资源型随机事件

当前归类如下：

- `act1_res_leftover_medicine`
- `act1_res_hidden_stash`
- `act1_res_food_shortage`

特点：

- 偏向资源涨跌
- 现在更像“轻量收益/损耗事件”

后续应补：

- 更多检定分流
- 不是固定拿或固定亏，而是根据属性/状态决定收益还是损失

### 3.2 关系型随机事件

当前归类如下：

- `act1_rel_peer_small_talk`
- `act1_rel_steward_cold_gaze`
- `act1_rel_false_friendliness`

特点：

- 主要影响人物印象
- 现在是轻关系扰动

后续应补：

- 更明确的“正向关系 / 负向关系”分支
- 检定后不同走向

### 3.3 异常型随机事件

当前归类如下：

- `act1_ano_hear_well_echo`
- `act1_ano_wrong_name_whisper`
- `act1_ano_mark_on_wall`

特点：

- 偏氛围、偏污染、偏异常
- 已经能提供部分线索

后续应补：

- 更明确的风险收益二选一
- 例如“吃污染换线索”或“压下异常保状态”

### 3.4 风险型随机事件

当前归类如下：

- `act1_risk_followed_in_dark`
- `act1_risk_patrol_question`
- `act1_risk_wrong_time_wrong_place`

特点：

- 主要给暴露、警惕、损失
- 是当前负面随机事件主体

后续应补：

- 不只是固定负面
- 应增加“检定成功则化险为夷并反拿收益”的结构

### 3.5 当前结论

随机事件这条线已经存在，但还不够统一。

现在的随机事件更像：

- 一组氛围/资源/风险扰动

而不是完整的：

- “随机出现 -> 检定 -> 根据表现吃正负结果”

所以后续这类事件应该优先统一成：

- 随机触发
- 至少一个检定
- 成功 / 失败明确分流

## 4. 奖励事件

定义：

- 核心作用就是给奖励
- 不以心战为核心
- 不以高风险为核心

当前项目里，这一类目前是最混杂的，因为很多奖励被塞进了“纯对话”或“条件事件”里。

### 4.1 已经基本可视为奖励事件的内容

当前可归入这一类的有：

- `conditional_record_discovery`
- `1301`
- `1302`
- `1303`
- `2301`
- `2302`
- `2303`
- `2401`
- `2402`
- `2403`

理由：

- 它们主要是在给线索、给路线推进、给状态
- 不依赖心战胜负
- 玩家体验更接近“推进后拿到实质收益”

### 4.2 目前应被改造成奖励事件的纯对话

下面这些虽然当前是对话事件，但从作用看，应该明确归为“奖励事件”：

- `2002`
- `2102`
- `2201`
- `2202`
- `2203`
- `2007`

理由：

- 它们主要负责交代新信息
- 交代完后本应稳定发放收益
- 非常适合做成“信息 + 线索 / flag / 经验 / 卡牌”

建议：

后续统一规定：

- 纯对话事件如果没有心战，就默认归入奖励事件

---

## 5. 回顾事件

定义：

- 用来恢复玩家状态
- 调整节奏
- 升级或整理牌组
- 给玩家一次“消化当前局势”的喘息机会

这类事件现在几乎还没正式存在。

### 5.1 当前能勉强算回顾的内容

严格说，当前几乎没有真正合格的回顾事件。

只有少量内容勉强带一点这个味道：

- `1102`
- `1103`
- `1104`

但它们本质上仍然是主线桥接，不是回顾事件。

### 5.2 当前缺失点

现在项目明显缺：

- 休整事件
- 恢复事件
- 牌组整理事件
- 升级卡牌事件
- 低风险的阶段回顾事件

### 5.3 后续应该补什么

建议新增一整类回顾事件，例如：

- 夜后调息
- 丹房外偷歇
- 茅屋整理遗物
- 复盘柳飞霞暗报
- 用异声或玉简强化一张卡

这一类的意义很大：

- 不让游戏一直只剩推进和受压
- 给牌组成长一个非战斗入口
- 让玩家能主动整理构筑

---

## 6. 商店事件

定义：

- 通过符合世界观的包装，让玩家进行卡牌购买、替换、删减、强化

当前项目里，这类事件完全没有。

### 6.1 为什么现在必须补

因为现在已经有：

- 牌库
- 卡牌奖励
- 心战构筑成长

如果没有商店事件，卡组成长会变成：

- 只能打赢才能涨
- 只能被动拿牌

这样会太窄。

### 6.2 符合当前背景的商店包装

这一类不能直接叫“商店”。

更适合的包装是：

- 黑市摊口
- 外门私换
- 药房暗柜
- 血役赌局
- 夜间偷售
- 旧物换命

### 6.3 商店事件应提供的功能

至少应该支持：

- 买牌
- 卖牌
- 删牌
- 换牌
- 强化牌

建议先做最小版本：

- 买牌
- 删牌
- 强化 1 张牌

### 6.4 当前归类

当前这一类：

- `暂无现成事件`

也就是说：

- 商店事件是后续必须新增的大类

## 8. 后续建议的落地顺序

### 第一批

先把现有“纯对话但实际应该发奖励”的事件正式视作奖励事件来补收益。

重点：

- `2002`
- `2102`
- `2201`
- `2202`
- `2203`
- `2007`

### 第二批

扩普通战斗池。

重点：

- 外门低级弟子
- 血役
- 替班巡夜

### 第三批

新增回顾事件。

重点：

- 恢复
- 删牌
- 强化
- 小幅整理构筑

### 第四批

新增商店事件。

重点：

- 先做“黑市摊口 / 私换 / 药房暗柜”之一
- 功能先做买牌、删牌、强化

---

## 9. 一句话结论

按你现在这套分类方式来看，当前项目最成熟的是：

- 对话战斗事件

当前最需要补的是：

- 奖励事件的规范化
- 回顾事件
- 商店事件

而随机事件这条线不是没有，而是还需要进一步统一成“随机触发 + 检定分流 + 正负结果”的标准形态。
