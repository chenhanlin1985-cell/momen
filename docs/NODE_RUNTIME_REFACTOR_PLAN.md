# 节点制运行时改造方案

## 目标

把当前“白天自由行动 + 手动移动 + 手动结束白天”的运行方式，改成“节点完成后自动推进下一节点”的结构。

这次改造的目的不是简化界面，而是让系统和内容表达重新对齐：

1. 玩家只在真正有价值的节点上做选择。
2. 主线、条件、随机三层结构继续保留。
3. 地点和人物更多承担叙事表现作用，而不是伪自由入口。
4. 运行时不继续增加兼容层和过渡层。

## 必须遵守的项目规范

### 1. 数字命名优先

运行时主键继续优先使用数字：

- 事件
- 节点组
- 人物
- 地点
- 动作

如果旧英文命名和数字命名并存，数字版是现行真源，英文版只允许被清理，不允许重新扩写。

### 2. 玩家文本继续全部外置

不能把这些内容写回脚本：

- 节点标题
- 节点正文
- 选项文案
- 线索描述
- UI 提示

它们继续放在 `.json / .csv / .md`。

### 3. 不保留半旧半新的混合状态

一旦某一层玩法已经切到节点制，对应旧入口就应当退役：

- 不能一边说系统自动推进，一边还保留“结束白天”。
- 不能一边说节点决定地点，一边还保留手动移动。
- 不能一边说节点直开 NPC，一边还保留自由点 NPC 作为主路径。

### 4. 不为兼容而发明新中间层

不新增这类东西：

- 过渡 action
- 伪 location
- 伪 day_end
- 临时桥接入口

能直接改主路径，就不包一层适配器。

## 新运行时定义

## 节点

节点是新的最小推进单位。

每个节点只负责一件事：

1. 播放一段必然剧情
2. 插入一个条件事件
3. 给出一个随机三选一
4. 进入结局收束

节点完成后，系统立刻解析下一节点，不再要求玩家手动“结束白天”。

## 节点解析顺序

每次节点结算后，统一按下面顺序解析下一步：

1. 主线必然节点
2. 条件节点
3. 随机三选一节点
4. 下一时间步或结局节点

这就是新的主循环。

## 时间步

保留“第几天”的外层壳，但内部不再是自由行动阶段，而是“某一天的第几个节点”。

建议最小结构：

- `day`
- `step`
- `lane`

其中：

- `day` 用于剧情节奏和结局条件
- `step` 表示当天第几个节点
- `lane` 表示当前节点层级
  - `main`
  - `conditional`
  - `random`
  - `ending`

## 退役或降级哪些系统

### 1. 手动移动

这些能力应当退役或降级：

- [location_service.gd](/C:/momen/systems/location/location_service.gd) 中的自由移动语义
- [run_controller.gd](/C:/momen/autoload/run_controller.gd) 中基于玩家点击地点的推进路径
- 地点切换弹窗和底栏地点入口

地点保留，但语义改成：

- 节点背景
- 节点舞台
- 条件标签载体

### 2. 手动结束白天

这条能力应直接退役：

- [run_controller.gd](/C:/momen/autoload/run_controller.gd) 中的 `end_day`
- 主界面上的“结束白天”按钮

### 3. 自由行动点

这条能力应降级成随机节点候选：

- [action_service.gd](/C:/momen/systems/action/action_service.gd)
- [action_definitions.json](/C:/momen/content/actions/action_definitions.json)

行动不再是“地点里的自由按钮”，而是“随机节点池给出的 3 个候选之一”。

### 4. 自由点击 NPC 触发主剧情

这条能力不再作为主路径保留：

- [npc_interactions.json](/C:/momen/content/npcs/npc_interactions.json)
- [npc_definitions.json](/C:/momen/content/npcs/npc_definitions.json)

主剧情人物应改成：

- 节点直开
- 节点指定出场
- 节点内部直接进入对话或事件

## 地点系统的新定位

地点不再是玩法系统，而是表现系统。

保留地点只有 3 个理由：

1. 给节点提供背景图和舞台氛围
2. 标记当前节点发生在哪里
3. 继续支撑少量地点相关条件判断

因此地点定义可以保留，但不再把它当成“玩家自由前往的地图菜单”。

## 动作系统的新定位

动作不再是“场景底部自由点选按钮”。

以后动作只有一种主要用法：

- 作为随机节点阶段的三选一候选项

也就是：

1. 当前没有主线必然节点
2. 当前没有条件节点
3. 系统从随机池生成 3 个可选节点
4. 玩家选 1 个
5. 立即结算并进入下一节点

## NPC 系统的新定位

NPC 不再作为“场景里随时可点的主剧情入口”，而是作为节点内容来源。

规则应当变成：

1. 如果当前节点是柳飞霞节点，就直接进入柳飞霞对话。
2. 如果当前节点是疯长老节点，就直接进入疯长老对话。
3. 如果当前节点是王麻子节点，就直接进入对应事件。

这样可以直接消除几类现有问题：

- 玩家要先移动到某地再点 NPC
- 对话结束后还要回自由行动层
- 对话结束后又被自动调度锁回别的事件

## 三层事件如何迁移

### 主线必然事件

继续作为主骨架使用。

现在的 `fixed_story` 可以直接迁移成“必出节点”。

### 条件事件

继续作为插入层使用。

它不再依赖“玩家刚刚在某地做了什么动作”，而是主要依赖：

- 当前天数
- 当前步数
- 标记状态
- 资源变化
- 前一节点的结果

### 随机事件

继续保留，但语义改变。

它不再是玩家行动后被动弹出的附加事件，而是“当前节点阶段可选的 3 个候选内容”。

## 新的主循环示例

1. 进入新局
2. 自动执行开场节点
3. 自动执行第一主线节点，例如柳飞霞初见
4. 节点结算后检查：
   - 有主线：直接进入下一主线
   - 无主线但有条件：插入条件节点
   - 都没有：给出随机三选一
5. 玩家完成当前节点
6. 系统立即进入下一节点
7. 达到结局条件后进入最终收束节点

## UI 改造原则

### 应保留

1. 当前剧情标题
2. 背景与出场人物
3. 对话面板
4. 线索模块
5. 资源与风险摘要

### 应删除或降级

1. 地点按钮栏
2. 行动按钮栏
3. 结束白天按钮
4. 自由移动弹窗

### 应新增或强化

1. 当前节点信息
   - 第几天
   - 第几个节点
   - 当前节点类型
2. 下一步状态提示
   - 主线推进
   - 条件插入
   - 随机三选一

## 推荐的最小改造顺序

### 第一步：先去掉“结束白天”

这是风险最低、收益最高的一步。

规则改成：

- 当前节点做完后自动推进
- 不再要求玩家手动点结束白天

### 第二步：把“移动”降级为背景切换

规则改成：

- 当前节点决定地点背景
- 玩家不再主动切地点

### 第三步：把 NPC 互动改成节点直开

规则改成：

- 柳飞霞、王麻子、疯长老、巡夜弟子等都通过节点进入
- 不再要求先点人

### 第四步：把随机行动改成三选一节点池

规则改成：

- 没有主线、没有条件时，系统给出 3 个随机候选
- 玩家选 1 个后立即结算

### 第五步：收口主界面

规则改成：

- 主界面只保留真实有效的入口
- 不再伪装成轻度自由探索界面

## 明确不建议做的事

1. 不保留“还能移动，但移动基本没意义”的状态。
2. 不保留“还有行动点，但其实只有一个有效选项”的状态。
3. 不一边做节点制，一边继续往自由探索层补内容。
4. 不借重构重新引回新的英文骨架。

## 当前结论

这次改造是大改，但方向是对的。

项目的实际目标已经不是“自由行动小地图玩法”，而是：

**主线、条件、随机三层共同服务于节点推进叙事。**
