# 事件时段槽规则

本文档用于固定当前项目里不同事件应当挂在哪个运行时槽位，避免后续继续把主线推进、商店、回顾、随机扰动全部塞进同一个入口。

原则：
- 先定槽位职责，再补事件
- 不为了临时能触发，就把所有东西都塞回 `post_action`
- 主线桥接、系统节点、随机扰动必须分层

## 一、当前有效槽位

### `phase_entry`

用途：
- 阶段一进入就应该立刻结算的事件

适合承载：
- 主线固定事件
- 晨间/夜间的强制桥接
- 战后摘要结果
- 对话战斗首拍

当前代表事件：
- `1001-1006`
- `1101-1104`
- `9102/9103`
- `9202/9203`
- `9302/9303`
- `9402/9403`
- `2001`
- `2003`
- `2004`
- `2005`

不适合放：
- 商店
- 回顾
- 白天行动后才有意义的尾段事件

### `post_action`

用途：
- 玩家白天做完一个节点动作后，立即检查的剧情推进

适合承载：
- 分线持续推进
- 白天行动触发的条件剧情
- 白天随机扰动

当前代表事件：
- `1301`
- `2102`
- `2201-2203`
- `2301/2303`
- `2401/2403`
- `9501`
- `act1_res_*`
- `act1_rel_*`
- `act1_risk_wrong_time_wrong_place`

不适合放：
- 商店
- 回顾
- 需要专门露脸一次的系统节点

### `day_tail`

用途：
- 白天动作与白天剧情推进都处理完后，进入夜晚前的专用尾段

适合承载：
- 商店事件
- 白天尾段的特殊系统节点
- 不该与剧情推进抢优先级的日间福利节点

当前代表事件：
- `3401`
- `3402`

规则：
- 先于 `day -> night` 结算
- 若当前白天还有事件或战斗，则不进入 `day_tail`
- 只建议挂“系统型节点”

### `night_tail`

用途：
- 夜间事件处理完后，进入收束阶段前的专用尾段

适合承载：
- 回顾事件
- 夜间恢复节点
- 不应和夜间主线桥接抢入口的系统节点

当前代表事件：
- `3301`
- `3302`

规则：
- 先于 `night -> closing` 结算
- 若当前夜里还有事件或战斗，则不进入 `night_tail`
- 只建议挂“恢复 / 牌组整理 / 回顾”类节点

## 二、按事件类型推荐挂载

### 主线事件
- 默认放 `phase_entry`

### 紧凑桥接事件
- 晨间/夜间桥接放 `phase_entry`
- 白天推进型桥接放 `post_action`

### 对话战斗事件
- 首拍放 `phase_entry`
- 若是白天行动后自然衔接出的持续拍，可放 `post_action`

### 摘要结果事件
- 默认放 `phase_entry`

### 随机事件
- 白天随机放 `post_action`
- 夜间随机放 `phase_entry`

### 奖励事件
- 如果它是剧情奖励的一部分：
  放在原剧情节点自己的槽位
- 如果它是系统奖励节点：
  白天放 `day_tail`
  夜间放 `night_tail`

### 回顾事件
- 默认放 `night_tail`

### 商店事件
- 默认放 `day_tail`

## 三、当前项目的明确约束

后续新增事件时，优先按下面规则判断：

1. 这是剧情推进，还是系统节点  
   剧情推进优先 `phase_entry / post_action`  
   系统节点优先 `day_tail / night_tail`

2. 这个事件如果不出现，会不会让主线断掉  
   会断掉的，不要挂在 `day_tail / night_tail`

3. 这个事件是否应该“稳定遇到一次”  
   如果是，优先挂专用尾段槽，不要丢进一堆条件事件里靠抢优先级

## 四、不再允许的旧做法

后续不要再做这些事情：

- 把商店事件挂进 `post_action`
- 把回顾事件挂进 `phase_entry`
- 把系统福利节点用极低优先级塞进剧情推进池
- 为了“能触发”临时把所有事件都改成 `phase_entry`

## 五、当前结论

目前项目的槽位分工已经可以稳定概括成：

- `phase_entry`：主线、强制桥接、摘要结果
- `post_action`：白天剧情推进、白天随机扰动
- `day_tail`：商店
- `night_tail`：回顾

后面补内容时，先看这张表，再决定事件挂哪里。
